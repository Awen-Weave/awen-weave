# Dolgellau Town Dataset — v0.1 Claim Schema

**Status:** draft for review. Supersedes v0 (`v0-schema.md`, kept for audit).
**Date:** 2026-05-10
**Driver:** the Tŷ Newyddion pilot (`pilot-ty-newyddion.md`) revealed six additive gaps in v0. v0.1 closes all six and resolves the two questions left open at v0 (predicate retirement, energy-study read pattern).
**Migration:** purely additive — every v0 claim remains valid under v0.1.

---

## 1. What's new from v0

Seven changes, in the order they appear in the schema below:

1. **`entity` gains a `visibility` column** — only meaningful for source entities (`public` | `restricted` | `private`). Read API filters source detail accordingly.
2. **`claim` gains three new value columns** — `value_entity_ref` (so claim values can point at other entities), `value_date_text` (human-readable original date expression like "c.1885"), and `qualifiers_json` (per-claim metadata like `name_type`, `floor_scope`, `date_precision`, `dialect`).
3. **Naming claims become multi-cardinality** with a required `name_type` qualifier (`current_local` / `listed_register` / `historic` / `vernacular`). The Tŷ Newyddion / Glyndwr Buildings / Newsagents case is now expressible.
4. **Three new entity types: `tenancy`, `event`, `research_question`** with their own predicate vocabularies. Tenancies handle occupancy timelines (Armitage → W H Smith → Wilkins → present); events handle refurbishments and designations; research questions promote known-unknowns to first-class records inside the Craidd.
5. **Entity-reference predicates** — `adjacent_to`, `contemporary_with`, `group_value_with`, `tenancy_of`, `affects_entity`, `relates_to_entity` — let the schema model the building-to-building relationships that listed-building "group value" depends on.
6. **`town` entity type** with a tightly-bounded starter predicate set (material tradition, street pattern, notable events, conservation authority, unitary authority, listed-building count, parish). Town-level facts are explicitly in scope where they bear on building histories; everything else stays out.
7. **Predicate deprecation mechanics** — predicates gain `deprecated_at` and `deprecation_reason`. Existing claims using a deprecated predicate stay valid; new claims using it are rejected. Closes the v0 open question.

Plus one operational decision:

8. **Energy-study read pattern: both API and nightly export.** A `GET /export/buildings.json` endpoint produces a public, signed snapshot every night, suitable for batch analysis (the energy study, future researchers). Live API stays the canonical query path. Closes the v0 open question.

## 3. Schema

### 3.1 `entity`

```sql
CREATE TABLE entity (
  entity_id     VARCHAR PRIMARY KEY,
  entity_type   VARCHAR NOT NULL,        -- see §3.4 for the controlled list
  uprn          BIGINT,
  toid          VARCHAR,
  visibility    VARCHAR,                 -- 'public' | 'restricted' | 'private'
                                         -- only meaningful when entity_type='source';
                                         -- null otherwise
  created_at    TIMESTAMP NOT NULL DEFAULT NOW(),
  notes         TEXT
);
```

Visibility semantics on the read API:

- `public` (default) — cite normally with title, organisation, URL, accessed_at.
- `restricted` — cite title and organisation only; URL and detailed evidence withheld; the read API returns "restricted source" markers in the place of detail.
- `private` — cite as "internal source" with the entity_id as the only externally-visible reference. Useful and honest: the public sees that *a* source exists and is recorded, just not what it is.

Internal Huw documents (the Tŷ Newyddion Operations Memo, the email to Coleg Cambria) become `visibility: private` source entities.

### 3.2 `claim`

```sql
CREATE TABLE claim (
  claim_id          VARCHAR PRIMARY KEY,
  subject_id        VARCHAR NOT NULL REFERENCES entity(entity_id),
  predicate         VARCHAR NOT NULL REFERENCES predicate(name),
  value_text        VARCHAR,
  value_int         BIGINT,
  value_real        DOUBLE,
  value_date        DATE,
  value_date_text   VARCHAR,                  -- NEW: "c.1885", "from 2016", "late C19"
  value_geom        GEOMETRY,
  value_cy          VARCHAR,
  value_en          VARCHAR,
  value_entity_ref  VARCHAR REFERENCES entity(entity_id),  -- NEW: for relationships
  qualifiers_json   VARCHAR,                  -- NEW: claim-level metadata, see below
  source_id         VARCHAR NOT NULL REFERENCES entity(entity_id),
  recorded_by       VARCHAR NOT NULL,
  recorded_at       TIMESTAMP NOT NULL DEFAULT NOW(),
  confidence        VARCHAR NOT NULL CHECK (confidence IN ('high','medium','low')),
  evidence_uri      VARCHAR,
  superseded_by     VARCHAR REFERENCES claim(claim_id),
  status            VARCHAR NOT NULL DEFAULT 'active'
                    CHECK (status IN ('active','superseded','disputed','withdrawn'))
);

CREATE INDEX claim_subject_predicate_idx ON claim(subject_id, predicate);
CREATE INDEX claim_status_idx ON claim(status);
CREATE INDEX claim_value_entity_ref_idx ON claim(value_entity_ref);
```

**Qualifiers** are claim-level metadata that aren't themselves claims. They describe *this* claim — its dialect, its applicable floor, its date precision, its name type — without warranting separate provenance. Controlled vocabulary in v0.1:

| Key | Domain | Values |
|---|---|---|
| `dialect` | bilingual claims with cy populated | `cy-GB-north` (default for Dolgellau) / `cy-GB-south` / `cy-GB-mid` / `cy-GB` (unspecified) / other ISO codes |
| `name_type` | claims on `name_cy` / `name_en` | `current_local` / `listed_register` / `historic` / `vernacular` |
| `floor_scope` | tenancy and event claims | `whole` / `ground` / `first` / `upper` / `attic` / `basement` / free-text where needed |
| `date_precision` | claims with `value_date` | `exact` / `year` / `decade` / `century` / `range` |

Why qualifiers and not predicates: a dialect is not a fact about the *building*. It's a fact about *this claim about the building*. Promoting them to predicates would make every claim bilingual into two claims, doubling the row count without adding provenance value. Qualifiers travel with the claim they describe.

### 3.3 `predicate` — with deprecation

```sql
CREATE TABLE predicate (
  name                 VARCHAR PRIMARY KEY,
  value_type           VARCHAR NOT NULL CHECK (value_type IN
                       ('text','int','real','date','geom','bilingual','entity_ref')),
  cardinality          VARCHAR NOT NULL CHECK (cardinality IN ('single','multi')),
  applies_to_types     VARCHAR NOT NULL,          -- JSON array, e.g. '["building","tenancy"]'
  description_cy       VARCHAR NOT NULL,
  description_en       VARCHAR NOT NULL,
  constraint_json      VARCHAR,
  required_qualifiers  VARCHAR,                   -- JSON array, e.g. '["name_type"]'
  added_at             TIMESTAMP NOT NULL DEFAULT NOW(),
  added_by             VARCHAR NOT NULL,
  deprecated_at        TIMESTAMP,                 -- NEW: null while active
  deprecated_by        VARCHAR,                   -- NEW
  deprecation_reason   VARCHAR,                   -- NEW
  superseded_by_predicate VARCHAR REFERENCES predicate(name)
                                                  -- NEW: optional pointer to replacement
);
```

Adding a predicate is a deliberate act, recorded in Prawf with action `predicate_added`. Deprecation is also a Prawf-recorded act (`predicate_deprecated`). Existing claims under a deprecated predicate remain queryable; new claims using it are rejected at the write API. Deprecated predicates appear in `GET /predicates?include=deprecated` for tools that need to read history.

`required_qualifiers` lets the schema enforce, for example, that every `name_cy` claim carry a `name_type` qualifier in `qualifiers_json`. The write API validates this.

`applies_to_types` lets the schema enforce, for example, that `tenant_name` is only valid on `tenancy` entities, `material_tradition` only on `town` entities, etc. Wrong-type claims are rejected at the write API.

### 3.4 Entity types

v0.1 controlled vocabulary for `entity_type`:

| Type | Purpose | Notes |
|---|---|---|
| `building` | A physical building or substantial structure. | UPRN/TOID anchors where available. |
| `street` | A named street or street segment. | Dolgellau-relative. |
| `area` | A bounded area — conservation area, ward, parish. | Carries a polygon. |
| `town` | The town as a whole subject. | Single instance for v1: TDS-DOL-T-00001. |
| `tenancy` | An occupancy of a building (or floor) by a tenant over a period. | Has `tenancy_of`, `tenant_name`, `period_start`, `period_end`, `tenancy_type`, optionally `floor_scope`. |
| `event` | A dated thing that happened. | Designations, refurbishments, changes of use, sales, fires, floods, demolitions. |
| `research_question` | A known-unknown the dataset is tracking. | Has `question_text`, `relates_to_entity`, `status`, optionally `answered_by_claim`. |
| `source` | A citation. | Carries `visibility`. |
| `person` | A named individual relevant to a building's history. | Used sparingly in v1 (architect, notable owner, founder). |

`person` is included because the Armitage / W H Smith founder / W D Wilkins line in the Tŷ Newyddion record will eventually want named individuals. In v1 we mention `person` exists but only create them when an actual research question resolves. Don't pre-populate.

## 4. Bilingual handling — unchanged from v0

Two patterns: paired fields on a single claim (default) versus separate claims with parallel predicates. Welsh stored when known, null when unknown — honest coverage. Dialect default `cy-GB-north`; deviations tagged on the claim's qualifiers.

## 5. Read interface — v0 plus export

All v0 endpoints unchanged. Two additions:

```
GET /export/buildings.json     # nightly signed snapshot
GET /export/<entity_type>.json # per-type snapshots
```

The nightly snapshot is generated at 03:00 local time, signed with the curator key, and held for 30 days under `/srv/town-dataset/exports/`. The energy study (and any other batch consumer) pulls this file rather than hammering the live API. The signed digest is published in Prawf so anyone can verify a copy they hold matches what the Pi served.

```
GET /predicates?include=deprecated   # NEW — include retired predicates
GET /research-questions?status=open  # NEW — discover open research
GET /buildings/{id}/relationships    # NEW — adjacent_to, group_value_with, etc.
```

## 6. Write interface — unchanged from v0

Two-tier curator/contributor model with mTLS authentication. Proposal queue for all non-curator writes. Self-acceptance forbidden at the API layer. The proposal review CLI (`craidd-review`) is updated to handle the new entity types — accepting a tenancy proposal creates the tenancy entity and its claims atomically.

## 8. Decisions taken in v0.1

**Predicate retirement: soft-deprecation.** A predicate gets `deprecated_at`, `deprecated_by`, `deprecation_reason`, and optionally `superseded_by_predicate`. Existing claims using it remain valid and queryable. New claims using a deprecated predicate are rejected at the write API. Deprecated predicates appear in `GET /predicates?include=deprecated`. Migrating claims to a successor predicate is a deliberate curator action, logged in Prawf as a supersession chain.

**Energy-study read pattern: API plus nightly export.** Live API for ad-hoc and authenticated access; signed nightly JSON snapshot at `/export/*.json` for batch consumers. The energy study pulls the snapshot for its own pipeline runs and uses the live API for spot-checks. Export digests are recorded in Prawf so any consumer can verify integrity.

**Dates with imprecision: hybrid.** `value_date` holds a sortable canonical date (the earliest plausible interpretation by convention). `value_date_text` holds the original imprecise form ("c.1885", "late C19", "from 2016"). `qualifiers.date_precision` records the precision (`exact` / `year` / `decade` / `century` / `range`). The API surfaces `value_date_text` to humans and uses `value_date` for filtering and sorting. EDTF (Library of Congress Extended Date/Time Format) is the eventual destination but unnecessary for v1.

**Person entities: declared but not pre-populated.** `entity_type='person'` is in the controlled vocabulary so research-question answers (Armitage, the c.1885 architect, the Wilkins family) can land cleanly when discovered. We do not pre-create person entities or populate genealogical fields. v1 is buildings + histories; people enter only as the answers to specific research questions.

## 10. Still open — for v0.2

Three genuinely deferred questions. The third was added 2026-05-10 after a curator review of the Tŷ Newyddion pilot.

1. **Floor- and unit-level sub-entities.** v0.1 handles floors via the `floor_scope` qualifier on tenancy and event claims. That works for v1 buildings + histories. When residential conversions, listed-building consents at unit level, or per-flat tenancies become the focus, sub-entities (`entity_type='unit'` with a `unit_of` relationship) will be required.
2. **EDTF dates.** The hybrid approach in §8 is good enough for v1. When research-question answers start producing date ranges of forms the hybrid can't carry cleanly ("between 1898 and 1903 inclusive", "before 1914"), migrate `value_date_text` to EDTF and add an EDTF parser in the read API.
3. **`listing` as a distinct entity type from `building`.** v0.1 conflates the two: a building is what's physically there, a listing is what Cadw chose to record, and v0.1 treats them as the same kind of subject — claims about a building include `listed_id` claims that point at the register entry. This works *within* the rule "model the physical building, attach listings to it as multi-cardinality claims," but it breaks down on real cases:
    - **Tŷ Newyddion / "Glyndwr Milk Bar"** (Cadw 4938 + 4939) — one physical building, two listings. Modelled in v0 as two separate buildings, then corrected on 2026-05-10 to one building with two `listed_id` claims. The correction works but loses structure: there's no clean way to attach claims that belong specifically to one register entry rather than the other.
    - **Tanrallt / Adre on Mount Pleasant** — one original building divided into two ownerships, *only one Cadw listing*, and that listing describes features that are now in the *other* ownership. v0.1 has no way to express "this listing describes features that are not all in the entity it's attached to."
    - General principle: physical buildings and Cadw register entries are not one-to-one. One building can carry zero, one, or several listings. One listing can cover one, several, or partial buildings. Modelling listings as if they were buildings is structurally wrong.
    
    **v0.2 proposal:** introduce `entity_type='listing'`. A listing has its own claims (`listing_authority`, `listing_grade`, `listing_reason`, `listing_date`, `register_id`, `feature_described`, `description_text`, `archive_url`). A listing relates to one or more buildings via a `lists` predicate (entity_ref, multi). A listing's described features (sash details, materials, dormers, etc.) live on the *listing*, not on the building — this resolves the Tanrallt/Adre case where the listing describes features outside its associated entity. Buildings continue to carry physical claims (geometry, material, roof_type, storeys); listings carry register claims (grade, register_id, authority). The two are joined explicitly. Migration from v0.1 is mechanical: each `listed_id` claim becomes a new `listing` entity, and the building gains a `lists` relationship in reverse.

## 11. SQL DDL — full v0.1

```sql
-- DuckDB-compatible. SpatiaLite needs the GEOMETRY column adjusted.

CREATE TABLE entity (
  entity_id     VARCHAR PRIMARY KEY,
  entity_type   VARCHAR NOT NULL CHECK (entity_type IN
                ('building','street','area','town','tenancy','event',
                 'research_question','source','person')),
  uprn          BIGINT,
  toid          VARCHAR,
  visibility    VARCHAR CHECK (visibility IN ('public','restricted','private') OR visibility IS NULL),
  created_at    TIMESTAMP NOT NULL DEFAULT NOW(),
  notes         TEXT
);

CREATE TABLE predicate (
  name                    VARCHAR PRIMARY KEY,
  value_type              VARCHAR NOT NULL CHECK (value_type IN
                          ('text','int','real','date','geom','bilingual','entity_ref')),
  cardinality             VARCHAR NOT NULL CHECK (cardinality IN ('single','multi')),
  applies_to_types        VARCHAR NOT NULL,
  description_cy          VARCHAR NOT NULL,
  description_en          VARCHAR NOT NULL,
  constraint_json         VARCHAR,
  required_qualifiers     VARCHAR,
  added_at                TIMESTAMP NOT NULL DEFAULT NOW(),
  added_by                VARCHAR NOT NULL,
  deprecated_at           TIMESTAMP,
  deprecated_by           VARCHAR,
  deprecation_reason      VARCHAR,
  superseded_by_predicate VARCHAR REFERENCES predicate(name)
);

CREATE TABLE claim (
  claim_id          VARCHAR PRIMARY KEY,
  subject_id        VARCHAR NOT NULL REFERENCES entity(entity_id),
  predicate         VARCHAR NOT NULL REFERENCES predicate(name),
  value_text        VARCHAR,
  value_int         BIGINT,
  value_real        DOUBLE,
  value_date        DATE,
  value_date_text   VARCHAR,
  value_geom        GEOMETRY,
  value_cy          VARCHAR,
  value_en          VARCHAR,
  value_entity_ref  VARCHAR REFERENCES entity(entity_id),
  qualifiers_json   VARCHAR,
  source_id         VARCHAR NOT NULL REFERENCES entity(entity_id),
  recorded_by       VARCHAR NOT NULL,
  recorded_at       TIMESTAMP NOT NULL DEFAULT NOW(),
  confidence        VARCHAR NOT NULL CHECK (confidence IN ('high','medium','low')),
  evidence_uri      VARCHAR,
  superseded_by     VARCHAR REFERENCES claim(claim_id),
  status            VARCHAR NOT NULL DEFAULT 'active'
                    CHECK (status IN ('active','superseded','disputed','withdrawn'))
);

CREATE INDEX claim_subject_predicate_idx ON claim(subject_id, predicate);
CREATE INDEX claim_status_idx ON claim(status);
CREATE INDEX claim_value_entity_ref_idx ON claim(value_entity_ref);

CREATE TABLE prawf_log (
  log_id         VARCHAR PRIMARY KEY,
  ts             TIMESTAMP NOT NULL DEFAULT NOW(),
  actor          VARCHAR NOT NULL,
  action         VARCHAR NOT NULL,
  target_id      VARCHAR NOT NULL,
  payload_json   VARCHAR NOT NULL,
  prev_hash      VARCHAR,
  this_hash      VARCHAR NOT NULL
);

CREATE VIEW current_claim AS
  SELECT * FROM claim
  WHERE status IN ('active','disputed')
  AND superseded_by IS NULL;

-- Coverage metric for public read API
CREATE VIEW cy_coverage AS
  SELECT
    p.name AS predicate,
    COUNT(*) FILTER (WHERE c.value_cy IS NOT NULL) AS cy_populated,
    COUNT(*) AS total,
    100.0 * COUNT(*) FILTER (WHERE c.value_cy IS NOT NULL) / NULLIF(COUNT(*),0) AS pct
  FROM claim c
  JOIN predicate p ON p.name = c.predicate
  WHERE p.value_type = 'bilingual'
  AND c.status = 'active'
  GROUP BY p.name;
```
